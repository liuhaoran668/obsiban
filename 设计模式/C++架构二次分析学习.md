开启canmanager ---> 订阅功能模块的回调函数（一直运行） ---> 回调函数处理信息逻辑 ---> 回调函数中通知waiter进行获取can发送过来的信息
### 从anglemanager进行分析：各个模块耦合进来
#### 一：职责分离，将功能独立出来，使用**组合模式**

在一个类中，让其负责的任务单一化，将对应的东西进行拆分实现。比如说手的速度控制，角度控制。添加回调函数。**通过can的订阅机制，由各自去处理信息。**

只要canmanager订阅了功能模块中的回调函数就会一直进行处理。收发是一直进行的，
- 在功能模块中要考虑多线程处理的问题。
	- 在这里面就使用了一个**waiters的对象进行管理订阅信息**，通过有信息就通知的模式。这样做的好处是避免cpu轮询和忙等去查询**信息是否过来。**
	- 一定要处理好边界问题，对于对象的析构或者close等逻辑都要考虑到多线程同时进行操作的问题。在这里面就是**关闭流式访问的上锁问题。确保避免在stop stream的时候不会访问。** 
#### 二：将通信层独立出来，统一由canmanage进行管理收发。

通过**订阅机制**进行管理每一个流经can的信息，由统一的can对象进行处理每功能模块。统一让回调函数去各自的功能模块去处理信息。**canmanager只负责收发信息。**

- 在这个逻辑中需要处理的东西就是**取消订阅要进行检查**，采用的是快照模式进行处理订阅的功能回调。----这个就是UAF问题。。。。**use-after-free**。
	- 通过增加一个**订阅状态结构体**来进行判断是否在其他线程中已经取消了订阅，保证状态的一致性。--在每次处理订阅的时候都进行检查。。。
	- 还有就是socket通信的关闭状态，对于这样的多线程共享变量（状态不一致导致的行为不同），一定要增加**状态检查机制**，常见的可能就是一个线程要发送，但是另一个线程已经进行了关闭，导致了文件描述符为-1.

#### 三：一定要考虑关闭函数或者析构函数的逻辑，避免多线程同时执行。

对于手的close函数，使用原子操作进行约束，确保线程安全。
```
if (closed_.exchange(true, std::memory_order_acq_rel)) {
return;
}
```
上述的这个原子操作函数会将**多个同时执行的线程按照内存序进行*排序*执行**，一个线程执行之后，其他线程会读取这个线程执行之后的状态。不会出现多个函数同时执行。


#### 四：如果存在回调的话，考虑回调在哪个线程中执行，考没考虑阻塞的问题。
在当前程序中，canmanager订阅各功能模块。订阅后再消息到达后，会使用线程去处理这些回调函数，在之前的方案中是将其顺序处理回调，可能会引起**阻塞问题**，这样在实时性上可能就会出现问题。

这个也就对应到了ros中的一个执行器，SingleThreadedExecutor这个执行器就是单线程的执行器，会阻塞，MultiThreadedExecutor是一个多线程执行器，会有一个线程池，在线程池里面进行处理对应的回调函数。

#### 五：考虑生命周期检查机制

在当前项目中的一个特点就是所有的功能组件（模块）都是**公共暴露出来**，当执行close后，再次调用这个功能成员可能会产生未定义的行为。**功能模块manager里面没有生命周期检查机制**。

一个线程执行close，生命周期结束，可能另一个线程还可以执行angle.set等。

#### 多线程下的一些原理

现在的计算机都是多核结构的，完全可以处理多线程的一个任务，为什么还是会需要锁和竞态分析等一些措施。
1. 主要就是在计算机中，多核处理的是**逻辑算法，但是他们却是共享的同一个内存**。

在内存中会出现什么问题，其中一个线程读取了一个对象的状态，进行存储。然后另一个线程在处理逻辑的时候又改变了对象的状态。但是第一个线程已经读取了。故出现了**同时读写同一块内存导致的数据不一致。** 会出现大量的问题。
	在c++中就有这样的一个问题，比如说vector对象，一个线程中已经读取了其的容量，另一个线程却添加或者删除一个对象，**导致容量变更，第一个线程在使用vector对象时，最终导致迭代器崩溃**。。。。

### 在c++二次分析中所遇到的问题
1. 在 L6 的 close 函数中，close 变量属于布尔（bool）量。**没有锁或者原子操作**，而是在最后将bool变成true。

在多线程环境下，如果只是简单的布尔量，可能存在数据竞争（Data Race）的问题，从而导致多次执行 close 逻辑。两个线程前后执行close函数，导致多次执行里面的逻辑。

2. Use-After-Free 风险
这个风险是某个对象在使用后在某个线程中进行了释放。在当前的 C++ 分析中，这类问题出现在订阅之后：系统会将任务推入一个订阅队列，并在一个循环线程中处理这个回调函数。
如果我将这个回调取消订阅，回调线程中的拷贝对象可能仍在执行相关的回调逻辑，因此会出现 Use-After-Free（UAF）风险。

拷贝复制对象无法保证回调已经结束。---**（这个属于是快照分发方法，可以最大限度的处理并行任务。避免竞态。）**


3. 状态检查缺失
这个状态检查其实也出现在多线程调用的情况中。在多线程调用中，一个线程获得并需要使用一个对象，而在另一个线程中，我将这个对象关闭并改变了它的状态。

如果在前一个线程中没有检查该对象的状态就直接进行操作，会导致未定义的行为（Undefined Behavior）。

在这版 C++ 的处理当中，针对 Socket 的文件描述符（file descriptor），我在一个线程中处理该描述符并进行 write 发送，而另一个线程却将该文件描述符关闭（stop），导致其返回值为 -1。

在这种情况下，如果需要调用 write 函数的线程没有提前进行状态检查，就可能会产生未定义的行为。
